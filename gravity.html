ODAS

With this package, you have everything you need in order to install ODAS on your machine in a self-contained way. This document assumes you are starting up on AWS.

Prerequisites

There are very few prerequisites to install ODAS, but in order to create a cluster, you will need:

  1. A 64-bit Linux machine. We recommend Amazon Linux 2.
  2. The Docker dameon (dockerd) should not be running.
  3. You should not be running anything on port 53 (e.g. dnsmasq)
  4. You should not be running local-dns, as it is not compatible with Kubernetes.
  5. You should have at least 150GB of disk space available in an EBS volume, mounted at /.
  6. Instance type should be t3.2xlarge, m5.2xlarge (or larger), c5.4xlarge (or larger). For functional validation, t3.2xlarge is recommended.

Security Group

We recommend you create a new dedicated Security Group for all the nodes, which we will call odas-sg.

For outgoing connections, it should allow all connections on all ports.

For incoming connections (IPv6 connections elided for clarity):

Type Protocol Port Range Source Description
SSH TCP 22 SSH
Custom TCP Rule TCP 8083 ODAS Web UI
Custom TCP Rule TCP 8089 ODAS REST API
Custom TCP Rule TCP 12050 ODAS Planner API
Custom TCP Rule TCP 12051 ODAS Planner Diagnostics
Custom TCP Rule TCP 13050 ODAS Worker API
Custom TCP Rule TCP 13051 ODAS Worker Diagnostics
Custom TCP Rule TCP 14050 ODAS JDBC API
Custom TCP Rule TCP 32009 ODAS Diagnostics
All traffic All All (e.g. odas-sg) ODAS SG Internal Traffic

NOTE: replace <sg id> with your SG identifier, e.g. sg-0655cc29afad577d3. NOTE: replace <admin cidr> with the CIDR range where you will do administrative tasks to the cluster, such as SSH or viewing diagnostic UIs. NOTE: replace <vpc cidr> with the CIDR range where other hosts that need to access the cluster are (e.g. EMR, web clients, etc) .

NOTE: By default, no processes are bound to ports 12051 and 13051 - but they are open to allow for diagnostics when needed.

Creating A New Cluster

Downloading

To create a cluster, you download the self-contained tar file that contains everything needed for the installation. You can find it here:

curl https://okera-release-uswest.s3.amazonaws.com/1.4.0-gravity/gravity/odas.tar --output

You can also use the quickstart install script which will download, unpack the tar file and run the preparation (you can replace uswest with useast or euwest as appropriate):

curl https://okera-release-uswest.s3.amazonaws.com/1.4.0-gravity/gravity/install.sh | sh -

Once they are all running, you can navigate to your machine’s IP/DNS at port 8083 to log into the UI. The default token of the cluster is root.

NOTE: this cluster is unauthenticated and does not have TLS enabled.

Enabling Authentication and TLS

By default, the cluster that starts up does not have any authentication enabled and is not using TLS. This is so you can initially test that it is functional and you can reach the cluster.

To enable authentication and TLS, you can run the following after the cluster is first brought up:

./okctl update

This will create a few things:

  1. A public/private key pair and a system (JWT) token that is signed by them. You can then create more login tokens using the okctl tokens create command.

  2. A local CA and a self-signed certificate that will be used by the cluster for TLS (i.e. HTTPS).

  3. An LDAP-like environment, where the username/password/groups are stored in .auth/users.json. You can modify them using the okctl users create and okctl users delete commands.

By default, you can now login with the system token (okctl tokens show system) and a username/password combination of okera/okera.

NOTE: you should not rely on the generated certificates and users for a production deployment - instead, you should set your organizations certificates and connect your system to LDAP.

Expanding The Cluster

If you have more machines to add to the existing cluster, you can copy this package (you can either get all of odas.tar or just the gravity binary) to that machine. You will then use the join command that was output when you ran okctl prepare earlier, which you was saved to a file called join.cmd (on the machine where you initially created the cluster).

You can execute it like this (where 10.1.10.68 is the address of the original node):

$ ./gravity join --token=VvYfYHGzku --role=worker 10.1.10.68

NOTE: you can leverage the script created at join.asg.cmd to execute on another node as well, even when not running in an ASG.

Removing A Node From The Cluster

If you would like to remove a node from the cluster, on that you node you can just execute:

$ ./gravity leave
Please confirm removing ip-10-1-10-124 (10.1.10.124) from the cluster (yes/no):
yes
launched operation "2b9bbed5-f306-4462-a0bc-8e5b77bdca97", use 'gravity status' to poll its progress

Note that if this is the only node in the cluster, you will need to add the --force flag, as this will delete the cluster permanently.

Expanding with an ASG

On AWS, you can configure an ASG to scale your cluster appropriately (we’ll refer to your initial node as the master node). To do this, simply create an ASG (using a Launch Template) with the same basic settings you used for your initial node (e.g. same AMI, same disk, same Security Group and IAM roles). For the user data, put in the contents of the join.asg.cmd that was created when you first ran okctl prepare. It will look something like this:

#!/usr/bin/env bash

mkdir okera-1.4.0-gravity
cd okera-1.4.0-gravity
curl https://okera-release-uswest.s3.amazonaws.com/1.4.0-gravity/gravity/gravity --output gravity
chmod 755 gravity
sudo ./gravity join --token=kjwYSXAxuu --role=worker 10.1.10.68 --cloud-provider=generic

Now, whenever a node boots up in the ASG, it will automatically join the cluster. You can see the additional nodes by running kubectl get nodes on your master node.

Removing ASG Nodes

ASGs do not provide a built-in way to run commands when they shut down in a reliable manner. This means that as nodes in the ASG get terminated, the cluster will think they should still be present and they will show up as offline. To remove them, on your master node, you can run:

sudo gravity remove <ip of node that is offline> --force

The --force is necessary since the node is already offline and cannot participate in the handshake.

User and Token Management

okctl has user and token management utilities with the okctl users and okctl tokens commands:

  tokens create [<flags>] <username> [<groups>...]
    Create a new JWT token for a specific username.

  tokens show <username>
    Show a token for a specific username.

  users create [<flags>] <username> <password> [<groups>...]
    [non-production only] Create a new user.

  users delete <username>
    [non-production only] Delete a user
