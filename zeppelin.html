Configuring Zeppelin to Run Spark As a User

For EMR clusters, there are configuration steps required in order for queries to run as a specific user (using their token) using basic authentication. It is preferable to setup Zeppelin with LDAP integration instead of using basic authentication.

These instructions are intended to complement the existing EMR setup instructions.

EMR Cluster Creation

When creating an EMR cluster, it’s easiest to add some per-user settings in the cluster configuration. As an example, assume that we want to setup the EMR cluster to allow a user named sarah to use Zeppelin with Spark. During EMR setup, these values can be added to prepopulate the pertinent configuration files. NOTE: This is in addition to the other settings in the EMR setup instructions.

[
  {
  "classification":"hadoop-core-site",
  "properties":{
    "hadoop.proxyuser.zeppelin.hosts":"*",
    "hadoop.proxyuser.zeppelin.groups":"*",
    "hadoop.proxyuser.sarah.hosts":"*",
    "hadoop.proxyuser.sarah.groups":"sarah"}
  },
]

If this is not done during EMR cluster creation, then it must be added to the /etc/hadoop/conf/core-site.xml file on all EMR cluster nodes and then the EMR processes must be restarted (you can use /usr/lib/okera/restart-emr.sh to do this).

Add the zeppelin User to the okera Group

This will be added to the bootstrap in future releases, but for now, run

sudo usermod -a -G okera zeppelin

Zeppelin Configuration

  1. Enabling user accounts

Copy the shiro configuration template (included with Zeppelin) into the expected location.

sudo cp /etc/zeppelin/conf/shiro.ini.template /etc/zeppelin/conf/shiro.ini

then edit /etc/zeppelin/conf/shiro.ini to add the user/password(s).

NOTE: If passwords contain punctuation, you may need to consult the shiro users guide. We tested only with simple passwords.

  1. Configure the zeppelin user for password-less sudo

Add this line to /etc/sudoers (best to use visudo than to edit directly).

zeppelin ALL=(ALL) NOPASSWD: ALL

  1. Configure how Zeppelin impersonates users

Edit /etc/zeppelin/conf/zeppelin-env.sh to add this line

export ZEPPELIN_IMPERSONATE_CMD='sudo -H -u ${ZEPPELIN_IMPERSONATE_USER} bash -c

  1. Restart the EMR services

Do this by running /usr/lib/okera/restart-emr.sh . The restart is necessary for Zeppelin to pick up the prior configuration changes. This will enable new options in the Zeppelin UI.

  1. Update the interpreters configuration

In the Zeppelin UI, go into the interpreters configuration section. Update both the sh and spark interpreter configurations to run to as “per user” and in “isolated processes”. After doing that, a new check box “User Impersonate” appears. Check that box.

  1. Fix the Zeppelin Spark submit environment variable

The current template has some environment variables that don’t seem to be picked up as we’d expect. To correct for this, delete the paths that include <CPS> or {{PWD}} with the absolute paths to the files in questions. An example would be:

export SPARK_SUBMIT_OPTIONS="$SPARK_SUBMIT_OPTIONS —conf 'spark.executorEnv.PYTHONPATH=/usr/lib/spark/python/lib/py4j-src.zip:/usr/lib/spark/python/:/usr/lib/spark/python/pyspark.zip:/usr/lib/spark/python/py4j-src.zip' —conf spark.yarn.isPython=true"

  1. Restart the EMR services (including Zeppelin)

Again, use the /usr/lib/okera/restart-emr.sh

  1. On the EMR master node, enable password-less ssh

This is required for the user(s) running Zeppelin queries Ensure that Hadoop can ssh as that user. In our working example, the hadoop user should be able to successfully execute the following command:

ssh sarah@localhost

One way to accomplish this is to generate an ssh key-pair for the hadoop user and append the public key to the user’s ~/.ssh/authorized_keys file.

  1. Update the permissions on Zeppelin’s log directory

sudo chmod -R 777 /var/log/zeppelin
